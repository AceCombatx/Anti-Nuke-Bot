Anti-Nuke Protection

Real-time monitoring and automatic restoration against destructive actions. Anti-Nuke V5 actively monitors for suspicious activities such as rapid channel deletion/creation, role deletion/creation, and server name/image changes. Key Capabilities:

Quick Setup Guide

Get Anti Nuke Bot V5 protecting your server in under 5 minutes

1

Invite the Bot

Add Anti Nuke Bot V5 to your server. For optimal protection, it is highly recommended to grant **Administrator** permissions and ensure its role is positioned at the very top of your server's role hierarchy (just below the server owner). This ensures the bot has the necessary authority to manage and protect all other roles and members.

Invite Bot
2

Run Setup Wizard

Initiate the interactive setup wizard using the command below. This guided process will walk you through essential settings, providing recommendations tailored to your server's size and needs. It's the easiest way to get started.

/setup-wizard
3

Configure Logging

Set up a dedicated, secure logging channel for all security logs, alerts, and audit trails. This is a **critical prerequisite** for all protection features to function correctly, as it provides essential audit trails and real-time alerts for detected threats.

/setup-logging #security-logs
4

Add Trusted Staff

Whitelist your trusted administrators and moderators to prevent false positives. Only whitelisted users (and the server owner, who is automatically whitelisted) can perform administrative changes without triggering protection measures. This is vital for smooth server management.

/whitelist add @Admin
5

Enable Protection Modules

Activate the core protection modules with their recommended settings. You can fine-tune these settings later using their respective configuration commands for granular control over each security aspect.

/antinuke-config enabled:true /antiraid-config enabled:true /antispam-config enabled:true /antiselfbot-config enabled:true /antialt-config enabled:true
6

Verify Setup

Confirm that all protection features are active and properly configured across your server. This command provides a quick overview of your bot's operational status and highlights any areas needing attention.

/protection-status
7

Create Initial Backup

Generate a comprehensive JSON backup file of all your bot's settings and configurations for the server. This is crucial for easy restoration or transfer of your setup. Ensure you store this file safely in a secure location.

/backup action:create

Pro Tip!

For a seamless onboarding experience, always start with the /setup-wizard command. It's designed to guide you through every critical configuration option with clear explanations.

If you encounter any issues, use the /errors command to access our comprehensive, interactive troubleshooting guide for immediate solutions and diagnostics.

Complete Command Reference

All commands available in Anti Nuke Bot V5, with detailed explanations, arguments, and examples.

Setup & Configuration

/setup-wizard
Purpose: Initiates an interactive, step-by-step setup process for the bot. This is ideal for new servers or for re-configuring the bot with guided assistance. It covers essential settings like logging, protection levels, whitelisting, and module activation.
Usage: Simply run the command and follow the prompts provided by the bot in your Discord server.
Notes: This command is highly recommended for first-time setup to ensure all critical configurations are addressed.
/setup-logging
Purpose: Designates a specific text channel for the bot to send all security logs, alerts, and audit trails. This channel is vital for monitoring bot activity and is a **required prerequisite** for all protection features to function correctly.
Arguments:
  • channel (required): The target text channel where logs will be sent. You can mention the channel directly (e.g., `#security-logs`).
Example: /setup-logging channel:#bot-logs
Notes: Choose a channel that only trusted staff can view to maintain security and privacy of logs.
/whitelist
Purpose: Manages a list of trusted users and roles who are exempt from the bot's protection measures. This is crucial to prevent legitimate administrative actions from triggering false positives. The server owner is always whitelisted by default.
Arguments:
  • action (required): Specifies the operation to perform:
    • add-user: Adds a user to the whitelist.
    • remove-user: Removes a user from the whitelist.
    • add-role: Adds a role to the whitelist.
    • remove-role: Removes a role from the whitelist.
    • list: Displays the current list of whitelisted users and roles.
  • user (optional, for user actions): The user to add or remove (e.g., `@AdminUser`).
  • role (optional, for role actions): The role to add or remove (e.g., `@ModeratorRole`).
Examples:
  • /whitelist action:add-user user:@TrustedMod
  • /whitelist action:remove-role role:@BotDevs
  • /whitelist action:list
Notes: Only whitelist users and roles you absolutely trust. Misuse of whitelisting can compromise your server's security.
/protection-status
Purpose: Provides a comprehensive overview of all active protection features, their current status (enabled/disabled), and key configuration parameters. This command helps you quickly verify that all systems are operational and correctly configured.
Usage: No arguments required.
Notes: Use this command regularly to monitor your bot's health and ensure all modules are running as expected.

Protection Configuration

/antinuke-config
Purpose: Configures the Anti-Nuke module, which protects against server destruction. You can set thresholds for various destructive actions and define the bot's response.
Arguments:
  • enabled (boolean): true or false to enable or disable the module.
  • channel_delete_threshold (integer): Maximum number of channel deletions allowed within a short period before action is taken (e.g., 3).
  • role_delete_threshold (integer): Maximum number of role deletions allowed before action (e.g., 2).
  • channel_create_threshold (integer): Maximum number of channel creations allowed before action (e.g., 5).
  • role_create_threshold (integer): Maximum number of role creations allowed before action (e.g., 3).
  • name_change_threshold (integer): Maximum number of server name/icon changes allowed before action (e.g., 1).
  • punishment (string): Action taken against detected nukers: ban or kick.
  • auto_restore (boolean): True or False to enable automatic restoration of deleted items (channels, roles).
  • webhook_protection (boolean): True or False to enable protection against malicious webhook creation/deletion.
  • invite_protection (boolean): True or False to enable protection against mass invite link generation.
  • emoji_protection (boolean): True or False to enable protection for server emojis.
  • sticker_protection (boolean): True or False to enable protection for server stickers.
  • thread_protection (boolean): True or False to enable protection for thread creation/deletion.
  • server_settings_protection (boolean): True or False to enable protection for critical server settings modifications.
Example: /antinuke-config enabled:true channel_delete_threshold:3 punishment:ban auto_restore:True webhook_protection:True
Notes: Adjust thresholds based on your server's activity. Lower thresholds provide stronger protection but may lead to more false positives if not properly whitelisted.
/antiraid-config
Purpose: Configures the Anti-Raid module to detect and mitigate mass joining events. You can define the rate at which new members join to trigger raid mode and the bot's response.
Arguments:
  • enabled (boolean): true or false to enable or disable the module.
  • join_rate (integer): Number of joins allowed within the `time_window` before triggering raid mode (e.g., 10).
  • time_window (integer): Time window in seconds to monitor join rate (e.g., 5).
  • punishment (string): Action taken against detected raiders: ban or kick.
  • auto_lockdown (boolean): True or False to enable automatic server lockdown during raids.
  • verification_boost (boolean): True or False to enable temporary verification level boost during raids.
  • new_account_threshold (integer): Minimum account age in days for new members to be considered suspicious (e.g., 7). Accounts younger than this threshold joining rapidly will be flagged.
Example: /antiraid-config enabled:true join_rate:10 time_window:5 punishment:kick auto_lockdown:True new_account_threshold:7
Notes: Consider your server's typical growth rate when setting `join_rate` and `time_window` to avoid false positives during legitimate member influxes.
/antispam-config
Purpose: Configures the Anti-Spam module to prevent various forms of message spam. You can set thresholds for message frequency and enable specific spam protections.
Arguments:
  • enabled (boolean): true or false to enable or disable the module.
  • message_threshold (integer): Number of messages allowed within the `time_window` before triggering action (e.g., 7).
  • time_window (integer): Time window in seconds to monitor message rate (e.g., 3).
  • punishment (string): Action taken against spammers: ban, kick, or mute.
  • mute_duration (integer): Duration of mute in minutes if punishment is `mute` (e.g., 10).
  • duplicate_detection (boolean): True or False to enable detection of repeated messages.
  • link_protection (boolean): True or False to enable protection against suspicious or unauthorized links.
  • caps_protection (boolean): True or False to enable protection against excessive use of capital letters.
  • emoji_spam_protection (boolean): True or False to enable protection against flooding channels with emojis.
Example: /antispam-config enabled:true message_threshold:7 time_window:3 punishment:mute mute_duration:10 duplicate_detection:True
Notes: Fine-tune `message_threshold` and `time_window` to suit your server's chat activity. Aggressive settings might impact legitimate conversations.
/antiselfbot-config
Purpose: Configures the Anti-Selfbot module to detect and respond to automated user accounts (selfbots). You can adjust the sensitivity of the detection.
Arguments:
  • enabled (boolean): true or false to enable or disable the module.
  • sensitivity (string): Detection sensitivity level: low, medium, or high.
    • low: Fewer false positives, may miss sophisticated selfbots.
    • medium: Balanced detection (recommended).
    • high: Catches more selfbots but has a higher false positive rate.
  • punishment (string): Action taken against detected selfbots: ban or kick.
Example: /antiselfbot-config enabled:true sensitivity:medium punishment:ban
Notes: Start with `medium` sensitivity and adjust based on observed selfbot activity and false positive rates.
/antialt-config
Purpose: Configures the Alt Account Detection module to manage and mitigate potential alternative accounts joining your server. You can set minimum account age requirements and define the bot's response.
Arguments:
  • enabled (boolean): true or false to enable or disable the module.
  • min_account_age (integer): Minimum account age in days required for a user to join the server (e.g., 7). Accounts younger than this will be flagged.
  • punishment (string): Action taken against detected alt accounts: ban or kick.
  • whitelist_verification (boolean): True or False to allow manual verification for borderline cases instead of immediate punishment. If enabled, users might be prompted for verification.
Example: /antialt-config enabled:true min_account_age:7 punishment:kick whitelist_verification:True
Notes: A higher `min_account_age` provides stronger protection but might inconvenience legitimate new users. Consider your community's typical new member profile.
/messagefilter-config
Purpose: Configures advanced message filtering rules, including custom word lists and link protection. This allows for granular control over unwanted content in your server's chat.
Arguments:
  • enabled (boolean): true or false to enable or disable the module.
  • punishment (string): Action taken for violations: delete, mute, kick, or ban.
  • invite_filter (boolean): True or False to enable filtering of Discord invite links.
  • link_filter (boolean): True or False to enable filtering of general suspicious or unauthorized links.
  • add_bad_word (string): A specific word or phrase to add to the custom bad words list.
  • remove_bad_word (string): A specific word or phrase to remove from the custom bad words list.
  • list_bad_words (boolean): True to display the current list of custom bad words configured for your server.
Examples:
  • /messagefilter-config enabled:true punishment:delete invite_filter:True link_filter:True
  • /messagefilter-config add_bad_word:spamword
  • /messagefilter-config list_bad_words:True
Notes: Use `add_bad_word` and `remove_bad_word` carefully. Regex patterns can be used for more complex filtering (consult documentation for syntax).
/antimention-config
Purpose: Configures the Anti-Mass Mention module to prevent excessive mentions of users, roles, or `@everyone`/`@here`. You can set thresholds and define the bot's response.
Arguments:
  • enabled (boolean): true or false to enable or disable the module.
  • mention_threshold (integer): Number of mentions allowed within a short period before triggering action (e.g., 5).
  • punishment (string): Action taken against mass mentioners: ban, kick, or mute.
  • mute_duration (integer): Duration of mute in minutes if punishment is `mute` (e.g., 15).
  • role_mention_protection (boolean): True or False to enable protection against mass role mentions.
  • everyone_protection (boolean): True or False to enable protection against `@everyone` and `@here` mentions.
Example: /antimention-config enabled:true mention_threshold:5 punishment:mute mute_duration:15 everyone_protection:True
Notes: Adjust `mention_threshold` based on your server's typical activity. Some communities use mentions frequently, so tailor settings to avoid false positives.

How accurate is the selfbot detection?

Our selfbot detection system is built on advanced behavioral analysis, utilizing multiple detection factors to achieve high accuracy. These factors include analyzing message patterns, inhumanly fast response speeds, automated command usage, and continuous 24/7 activity. The accuracy can be fine-tuned with configurable sensitivity levels:

  • Low Sensitivity: Designed to minimize false positives, but may allow more sophisticated selfbots to slip through. Recommended for servers with very active or unique user behavior.
  • Medium Sensitivity (Recommended): Offers a balanced approach, providing strong detection capabilities with a reasonable false positive rate. This is ideal for most Discord communities.
  • High Sensitivity: Maximizes selfbot detection, catching a broader range of automated accounts. However, this setting has a higher potential for false positives, which may require more manual review by server staff.

We recommend starting with medium sensitivity and adjusting based on your server's specific needs and activity patterns. Always manually verify detections before taking permanent punitive action against a user.

Why aren't the protection features working?

All protection features in Anti Nuke Bot V5 require proper setup and sufficient permissions to function effectively. If you're experiencing issues, please follow these troubleshooting steps:

  1. Run the Setup Wizard: Execute /setup-wizard for a guided, interactive configuration process that ensures all essential settings are correctly applied.
  2. Configure Logging: Ensure a dedicated logging channel is set up using /setup-logging #channel. This is absolutely critical, as all protection features rely on sending logs and alerts to this channel.
  3. Whitelist Trusted Admins: Add all your legitimate administrators and moderators to the whitelist using /whitelist add @Admin. Non-whitelisted administrative actions will trigger protection measures.
  4. Enable Modules Individually: Confirm that each protection module you intend to use is explicitly enabled via its respective configuration command (e.g., /antinuke-config enabled:true).
  5. Verify Permissions and Role Position: Ensure the bot has all required permissions (ideally Administrator) and that its role is positioned at the very top of your server's role hierarchy, just below the server owner. Incorrect role placement is a common cause of issues.
  6. Check Overall Status: Use /protection-status to get a quick overview of all active features and identify any modules that might be disabled or misconfigured.
  7. Use the Error Guide: For specific issues, run /errors to access our comprehensive, interactive problem-solving guide.

What permissions does the bot need?

For optimal protection and full functionality, Anti Nuke Bot V5 highly recommends and ideally requires **Administrator** permissions. Granting Administrator permissions ensures:

  • Comprehensive Monitoring: Full access to monitor all server activities, including audit logs, member actions, and guild updates.
  • Instant Remediation: The ability to instantly reverse malicious actions, such as deleting channels, recreating roles, or restoring server settings.
  • Full Control: Permissions to manage roles, channels, members (kick, ban, timeout), webhooks, and other critical server elements.
  • Conflict Prevention: Minimizes permission conflicts that could compromise the bot's ability to protect your server effectively.

Critical Importance of Role Position: Beyond permissions, the bot's role **must be positioned at the very top** of your server's role hierarchy (just below the server owner). This ensures the bot has the necessary authority to manage and protect all other roles and members, preventing lower-ranked malicious roles from bypassing its defenses.

You can use the in-bot command /errors and select "Check Permissions" to get a detailed report on any missing permissions that might be affecting the bot's performance.

How do I prevent false positives?

To minimize false positives and ensure legitimate administrative or user actions are not flagged by Anti Nuke Bot V5, consider the following best practices:

  • Whitelist Trusted Staff: Use the /whitelist add @User or /whitelist add @Role commands for all your legitimate administrators, moderators, and trusted bots. The server owner is automatically whitelisted.
  • Adjust Sensitivity: For modules like Anti-Selfbot, start with a lower sensitivity setting (e.g., /antiselfbot-config sensitivity:low) and gradually increase it as needed, observing the bot's behavior.
  • Configure Thresholds: Set appropriate limits for your server's size and activity level. For example, adjust the message_threshold for Anti-Spam or the join_rate for Anti-Raid to match your community's typical activity.
  • Utilize Warning Mode: Some features offer a "warn" punishment option as a first response. This allows you to receive alerts and manually review the situation before the bot takes harsher, automated actions.
  • Monitor Logs Regularly: Consistently review the detection logs in your configured logging channel. This helps you identify patterns of false positives and fine-tune your settings accordingly.
  • Periodic Review: Periodically use the respective -config commands or /protection-status to review and adjust your settings as your server evolves.

Can I customize the detection algorithms?

Yes! Anti Nuke Bot V5 offers extensive customization options for its detection algorithms and responses, allowing you to tailor the bot's behavior to your server's unique needs:

  • Sensitivity Levels: Adjust the detection sensitivity for various modules (e.g., `low`, `medium`, `high` for Anti-Selfbot) to balance protection and false positives.
  • Custom Thresholds: Set specific numerical limits for activities such as message rates, join rates, channel/role deletions, and more, ensuring the bot responds appropriately to your server's activity volume.
  • Punishment Escalation: Configure progressive punishment systems, from initial warnings and temporary mutes to kicks and permanent bans, allowing for flexible responses to violations.
  • Time Windows: Define custom monitoring periods for different activities to effectively catch both rapid bursts of malicious actions and slower, more coordinated attacks.
  • Whitelist Rules: Create complex whitelist configurations for individual users, specific roles, or even particular channels, ensuring trusted entities are not affected by protection measures.
  • Custom Filters: Add server-specific content filters for messages, including custom bad words and advanced regex patterns, to enforce your community's content guidelines.

You can access these customization options through the respective -config commands (e.g., /antinuke-config, /antispam-config) or utilize the interactive /setup-wizard for a guided customization experience.
